# SwarmCom - Next Steps & Feature Roadmap (March 2026)

Created based on code review with Grok on March 28, 2026.

## Current Strengths
- Excellent local-first quickstart (`npm run quickstart`)
- Matrix as reliable, federated, E2EE-capable backbone
- Hierarchical roles (boss / peer / worker) with signed messages
- Visibility-first design with STATUS.md and JSON snapshots
- MCP-native + OpenClaw support
- Lightweight and self-hosted by design

## Priority Gaps to Close (v0.1 → v0.2)

### 1. Persistence Layer
- [ ] Add SQLite database for message history and status timeline
- [ ] Store all messages with timestamps, sender, role, and payload
- [ ] Implement simple migration system
- [ ] Keep STATUS.md as the human-readable view (auto-generated from DB)

### 2. Authentication & Identity
- [ ] Add optional persistent node identities using ed25519 keypairs
- [ ] Support Matrix user accounts as fallback for federation
- [ ] Move beyond shared secrets for larger or cross-organization swarms

### 3. Hierarchical Summarization & Rollups
- [ ] Implement `query_subtree_status()` 
- [ ] Auto-generate clean hierarchical summaries for parent nodes
- [ ] Support "rollup every X minutes" or on-demand

### 4. Resilience & Observability
- [ ] Add heartbeat mechanism (every 30s)
- [ ] Implement reconnection logic and "node offline" status
- [ ] Add structured logging (`swarmcom.log`)
- [ ] Create `swarmcom status` CLI command showing live network health

### 5. Developer Experience (SDK)
- [ ] Create lightweight TypeScript client SDK (`swarmcom-client.ts`)
- [ ] Provide simple API:
  ```ts
  const swarm = new SwarmComClient(config);
  await swarm.sendMessage(roomId, { type: "task", payload: {...} });
  await swarm.subscribeToRoom(roomId, callback);
  ```
